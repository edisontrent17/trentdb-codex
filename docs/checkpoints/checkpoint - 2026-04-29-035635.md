## Goal

The project direction was formalized as a production-oriented analytical SQL database in Java. DuckDB is the architectural reference, while Postgres-like behavior is the semantic reference for the supported SQL subset.

The immediate implementation goal was to move from parser and planner scaffolding into a real execution path, then add a DuckDB-style replacement scan extension point so CSV files can be queried as native table-like sources:

```sql
SELECT * FROM 'people.csv';
```

The user explicitly rejected a `read_csv` function and rejected CSV-specific logical or physical scan operators. The accepted design was a generic replacement scan abstraction.

## Architecture

The current end-to-end query path is:

```text
SQL
 -> SqlParser / AstBuilder
 -> Binder
 -> LogicalPlanner
 -> PhysicalPlanner
 -> PipelineExecutor
 -> QueryResult
```

The architecture now follows the DuckDB-shaped pipeline:

- parser produces AST nodes
- binder resolves catalog names, replacement scans, functions, column ordinals, and types
- logical planner builds a relational logical operator tree
- physical planner lowers the tree into a push-based pipeline
- pipeline executor pushes vectorized `DataChunk` batches from source to sink

The project now has explicit subsystem boundaries for:

- catalog metadata
- type metadata
- transaction/snapshot API shape
- binding
- logical planning
- physical planning
- vectorized execution
- storage
- function binding/execution
- replacement scans

## Project Rules And Documentation

`AGENTS.md` was updated to define the project as production-oriented. The important rules now are:

- mirror DuckDB's subsystem boundaries where practical
- keep the pipeline aligned with parser -> binder -> logical plan -> optimizer -> physical plan -> execution
- preserve explicit catalog entries instead of collapsing metadata into ad hoc objects
- keep read paths and write paths explicitly separated
- route DDL and DML through write-aware boundaries that can later provide WAL, recovery, and transactional semantics
- require WAL design for writes before durable/committed visibility
- treat DDL as a write operation
- make reads snapshot-aware even though reads do not append WAL
- design catalog and table scans around transaction snapshots

The documentation also captures that early storage may remain in memory, but APIs should not assume global mutable catalog/table state can be read directly without visibility checks.

## Transactions And Snapshot Shape

The transaction layer is still early, but the API shape exists:

- `TransactionManager`
- `Transaction`
- `TransactionSnapshot`

Catalog operations and binder lookup receive a `Transaction`. That matters because it leaves a clear place to enforce MVCC catalog visibility later.

Current behavior is effectively single-threaded/trivial snapshot behavior. The important architectural point is that the code is no longer shaped around direct global access as the permanent contract.

## Logical Planning

- `LogicalGet`
- `LogicalFilter`
- `LogicalProjection`
- `LogicalExplain`

`LogicalGet` owns a `BoundTableRef`, not a CSV-specific or storage-specific object. This keeps the logical plan independent of whether the source is a catalog table or a replacement scan.

`LogicalPlanPrinter` prints explain output such as:

The physical model is push-based:

```text

```

`PhysicalPlanner` lowers:

- `LogicalProjection` -> `PhysicalProjection`

`QueryExecutor` now executes a planned `Pipeline` and returns the sink's `QueryResult`.

## Vectorized Execution

Execution is chunk-oriented and vectorized.

Core vector classes:

- `DataChunk`
- `Vector`
- `VectorType`
- `SelectionVector`
- `ValidityMask`

Supported vector types:

- `FLAT`
- `CONSTANT`
- `DICTIONARY`

`FilterOperator` / `PhysicalFilter` evaluate predicates into selection vectors. Only exact `TRUE` passes a filter; `FALSE` and `NULL` do not pass. Empty filtered chunks still preserve schema when forwarded.

`ExpressionExecutor` evaluates bound expressions over chunks, including:

- column references
- literals
- scalar functions
- comparisons
- `AND`
- `OR`

Boolean logic follows SQL three-valued behavior for the implemented cases.

## Storage

Storage is currently in memory:

- `StorageManager`
- `InMemoryTableStorage`
- `StorageException`

Catalog-backed `PhysicalTableScan` reads chunks from `StorageManager`. This is intentionally separated from catalog metadata so future persistent storage, MVCC row visibility, and WAL-backed writes can be introduced behind stable boundaries.

## Replacement Scan Architecture

The replacement scan subsystem was added to support DuckDB-style external source resolution without adding source-specific logical or physical operators.

Interfaces/classes:

- `ReplacementScanRegistry`
- `ReplacementScanProvider`
- `ReplacementScan`
- `ReplacementScanFunction`
- `CsvReplacementScanProvider`

Runtime flow for:

```sql
SELECT name FROM 'people.csv' WHERE id = '1';
```

is:

```text
AstBuilder
  string literal FROM -> TableReference.path(...)

Binder
  TableReference.isPath()
  -> ReplacementScanRegistry.replace(path)
  -> CsvReplacementScanProvider.tryReplace(path)
  -> BoundTableRef.replacement(...)

LogicalPlanner
  -> LogicalGet(BoundTableRef)

PhysicalPlanner
  -> PhysicalTableScan(BoundTableRef)

PhysicalTableScan
  if catalog table: StorageManager.getTable(...).scanChunks()
  if replacement scan: replacementScan.scanFunction().scan()

PipelineExecutor
  -> filter/projection/result sink
```

CSV-specific code is limited to `CsvReplacementScanProvider`. There is no `LogicalCsvScan` and no `PhysicalCsvScan`.

Current CSV behavior:

- recognizes `.csv` paths
- reads local files
- requires a header row
- maps all columns to `LogicalType.TEXT`
- returns data as `DataChunk`
- treats empty field values as `NULL`
- uses simple comma splitting

Current CSV limitations:

- no full RFC 4180 handling
- no quote-aware parser
- no type inference
- no projection pushdown
- no filter pushdown
- no compressed or remote files
- no catalog registration

## Current Limitations

The system is still early and intentionally limited:

- writes are not yet WAL-backed
- MVCC visibility is API-shaped but not fully implemented
- storage is in-memory
- optimizer phase is not implemented
- joins are parsed but not bound/executed
- `GROUP BY`, `ORDER BY`, and `LIMIT` are parsed but rejected by binder
- scalar function support is minimal
- CSV parsing is basic
- replacement scans do not push projection or filters
- there is no persistent catalog
- there is no crash recovery

## Next Steps

The next logical implementation steps are:

- harden CSV replacement scan behind the same generic interface with a real quote-aware CSV reader
- add file/table function style option plumbing only if it can preserve the replacement scan abstraction
- add projection pushdown to replacement scans without making CSV special in the planner
- add filter pushdown later through generic scan capabilities
- introduce an optimizer boundary between logical and physical planning
- expand physical execution operators while keeping the push pipeline model
- start write-path architecture with WAL record interfaces before implementing durable DDL/DML
- deepen MVCC snapshots so catalog and table scans use visibility rules consistently
