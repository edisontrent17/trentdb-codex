Correlated Scalar Aggregate Subqueries as SINGLE Joins

This PR replaces the worst correlated scalar aggregate execution path with a join-shaped plan. Instead of rerunning the aggregate subquery once per outer row, TrentDB scans the inner table once, builds an aggregate lookup keyed by correlated columns, then probes that lookup for each outer row.

Primary target TPC-H Q20 Canonical correlated scalar aggregate shape.
Before recorded smoke time 64,335 ms Per-row correlated subquery execution.
Current PR smoke time 125 ms Native aggregate lookup plus probes.

The Problem

Before: execute scalar subquery per outer row

for each candidate partsupp row:
    scan lineitem
    apply l_partkey = ps_partkey
    apply l_suppkey = ps_suppkey
    apply date filter
    compute sum(l_quantity)
    compare ps_availqty > 0.5 * sum

This is correct, but it repeats the same inner scan many times. Q20 exposes that cost clearly.

After: build once, probe many times

scan lineitem once:
    apply date filter
    group by (l_partkey, l_suppkey)
    compute sum(l_quantity)

for each candidate partsupp row:
    lookup (ps_partkey, ps_suppkey)
    compare ps_availqty > 0.5 * sum

This is the narrow first step toward DuckDB-style decorrelation.

Pipeline Shape

SQL Correlated scalar aggregate in WHERE.
Binder Marks outer references and scalar subquery metadata.
Logical Planner Rewrites eligible scalar subquery to dependent join kind SINGLE.
Physical Planner Maps SINGLE to PhysicalCorrelatedScalarAggregateJoin.
Execution Build aggregate hash table once, then append scalar marker values.

Q20 Shape

ps.ps_availqty > (
    SELECT 0.5 * sum(l.l_quantity)
    FROM lineitem l
    WHERE l.l_partkey = ps.ps_partkey
      AND l.l_suppkey = ps.ps_suppkey
      AND l.l_shipdate >= DATE '1994-01-01'
      AND l.l_shipdate < DATE '1995-01-01'
)

The equality predicates become lookup keys. The date predicates are residual filters applied while building the lookup.

Plan Change Summary

LogicalDependentJoin type=SINGLE
  marker=#scalar...

PhysicalSingleJoin subquery=SCALAR
  build keys: l_partkey, l_suppkey
  output marker: 0.5 * sum(l_quantity)

The final WHERE predicate compares the outer column with the appended scalar marker column.

Physical Plan
  Source: PhysicalTableScan table=supplier
  Operators
    PhysicalHashJoin / PhysicalNestedLoopJoin ...
    PhysicalSingleJoin subquery=SCALAR
      build:
        scan lineitem once
        filter shipdate range
        aggregate sum(l_quantity)
        key by (l_partkey, l_suppkey)
      output:
        append #scalar marker column
    PhysicalFilter
      predicate:
        ps_availqty > #scalar
    PhysicalOrder
    PhysicalProjection
  Sink: RESULT_COLLECTOR

The scalar aggregate is now a physical operator. It builds the lookup once and probes it for each outer row.

Eligibility Rules

Rule Why it exists Unsupported behavior
Subquery must be correlated. Uncorrelated scalar subqueries already have a separate execution path. Leave existing path unchanged.
Subquery FROM must be one table or replacement scan. The new physical lookup builds from a single inner scan. Leave existing evaluator path.
One ungrouped aggregate output. The lookup stores one scalar aggregate result per key. Leave existing evaluator path.
No DISTINCT aggregate yet. DISTINCT needs different aggregate state. Leave existing evaluator path.
One or two correlated equality keys. The hash table stores native long keys for the first slice. Leave existing evaluator path.
Key types: BOOLEAN, INTEGER, BIGINT, DATE. These map directly to native long keys without Object-backed keys. Leave existing evaluator path.
Correlated non-equality predicates are not rewritten. Those need dependent join residual handling or a later decorrelation rule. Leave existing evaluator path.

Semantic Details

Missing lookup rows are not blindly treated as SQL NULL. They are evaluated as an aggregate over an empty input, matching scalar aggregate semantics.

count(*) over no matching rows returns 0.
sum(...), avg(...), min(...), and max(...) over no matching rows return NULL.
A NULL outer correlation key also behaves like no matching inner rows.
The native key path does not use Long.MIN_VALUE as a null sentinel, so real BIGINT min keys remain valid.

Files To Review

LogicalPlanner: detects eligible scalar aggregate subqueries and rewrites them to LogicalDependentJoin.Kind.SINGLE.
LogicalDependentJoin: now represents both MARK and SINGLE dependent joins.
PhysicalPlanner: maps MARK to the existing exists mark join and SINGLE to the new scalar aggregate join.
PhysicalCorrelatedScalarAggregateJoin: scans the inner table, builds aggregate lookup state, and appends the scalar marker vector.
ScalarAggregateHashTable: native long-key aggregate hash table and scalar value materialization.
QueryExecutorTest: targeted regression tests for plan shape, fallback, empty groups, and key edge cases.
What this PR does not do: It does not implement arbitrary query unnesting yet. It is a production-safe first slice: only the scalar aggregate shapes the new physical operator can execute are rewritten. Everything else keeps the previous evaluator path.
